Introduction to Git and GitHub

The Problem Without Version Control

Meet Our Coding Team

Three friends from the Doraemon series have decided to build a School Project Tracker web application together:

Gian

Team Lead

Manages main codebase

Nobita

Frontend Developer

Works on user interface

Suneo

Backend Developer

Works on server logic

Problems Without Git

Initially, they try to manage their code by saving files on their computers and sharing via email or USB drives. Here are the problems they face:

Problem 1: Lost Work

Gian creates a file called app.txt with some code. Next day, he makes changes and saves the file again. Now the original version is gone forever. If the new code has bugs, he cannot go back to the working version.

Problem 2: Messy File Names

To avoid losing old versions, they start creating multiple copies:

  • app.txt
  • app_v2.txt
  • app_final.txt
  • app_final_REALLY_FINAL.txt
  • app_gian_fixed_v3.txt

This becomes impossible to manage!

Problem 3: Collaboration Chaos

Nobita emails his homepage.txt file to Gian. Suneo also emails his version of homepage.txt. Now Gian has two files with the same name but different content. Which one is correct? How to combine them?

Problem 4: No History

When a bug appears in the code, they cannot see who made what changes and when. Debugging becomes very difficult.

Problem 5: Overwriting Work

Gian and Nobita both edit styles.txt. When they share files, one person's work gets completely overwritten and lost.

Problem 6: No Backup

If Gian's laptop crashes or gets stolen, all the code is lost forever. There is no backup.

Without Git - The Chaos:

Gian's Computer:            Nobita's Computer:          Suneo's Computer:
app.txt                     app_nobita.txt              app_suneo.txt
app_v2.txt                  homepage.txt                server_code.txt
app_final.txt               homepage_new.txt            database.txt
app_FINAL2.txt              styles.txt                  
                            styles_updated.txt          

Problems:
- No sync between computers
- No history of changes
- No way to merge work
- Files sent via email/USB
- Easy to lose work

The Solution: Git

Git is a Version Control System that solves all these problems. It is a tool that tracks every change made to your files, who made the change, and when it was made.

How Git Helps:

  • Snapshots: Git saves complete snapshots of your project at different points in time
  • Time Travel: You can go back to any previous version instantly
  • Collaboration: Multiple people can work on the same files without conflicts
  • History: See exactly who changed what, when, and why
  • Branching: Work on new features without breaking the main code
  • Backup: Store code safely in the cloud using GitHub

Git vs GitHub

Many beginners confuse Git and GitHub. They are different things:

Git is a program you install on your computer. It tracks changes to your files locally on your machine.

GitHub is a website (github.com) where you can upload and store your Git projects online. Think of it like Google Drive but specifically designed for code.

Simple Analogy:

  • Git is like your personal notebook where you write your work and keep every draft
  • GitHub is like a library where you store a copy of your notebook so others can read it and contribute

You can use Git without GitHub (working locally only). But GitHub needs Git to function. In this guide, you will learn both.

Next: Installing Git

Installing Git

Before you can use Git, you need to install it on your computer. The installation process is different for Windows, Mac, and Linux.

Installing on Windows

  1. Open your web browser and go to https://git-scm.com
  2. Click the big Download for Windows button
  3. Once the file downloads, run the installer (it will be named something like Git-2.43.0-64-bit.exe)
  4. Click Next on all the screens - the default settings are perfect for beginners
  5. Click Install and wait for it to finish
  6. Click Finish when done

After Installing on Windows

You now have a program called Git Bash. This is a special terminal window where you will type all your Git commands.

To open it: Click the Start Menu and search for "Git Bash"

Installing on Mac

Mac computers often come with Git pre-installed. Let's check:

  1. Open the Terminal application (press Cmd + Space, type "Terminal", and press Enter)
  2. Type this command and press Enter:
git --version

If you see a version number like git version 2.39.0, Git is already installed!

If Git is not installed, your Mac will automatically show a popup asking you to install it. Click Install and follow the prompts.

Installing on Linux

Open your terminal and run these commands:

For Ubuntu/Debian:

sudo apt update
sudo apt install git

For Fedora:

sudo dnf install git

Verifying Installation

After installing, let's verify that Git is working correctly:

  1. Open your terminal (Git Bash on Windows, Terminal on Mac/Linux)
  2. Type this command and press Enter:
git --version

You should see output like:

git version 2.43.0

If you see a version number, congratulations! Git is installed and working.

Important Note About Commands

In many tutorials, you will see commands written with a dollar sign like this: $ git --version

Do not type the dollar sign! It just means "this is a terminal command". Only type what comes after it.

First-Time Setup

Before using Git for the first time, you need to tell it your name and email address. This is important because every change you make will be recorded with your name.

Why is This Needed?

When you save a snapshot of your code (called a "commit"), Git records:

  • What changed
  • When it changed
  • Who made the change

This is very important when working in a team. Everyone can see who did what and when.

Setting Your Name

Open your terminal (Git Bash on Windows) and type this command. Replace "Gian" with your actual name:

git config --global user.name "Gian"

Press Enter. You will not see any output - that is normal.

Setting Your Email

Now set your email address. Use the same email you will use for your GitHub account:

git config --global user.email "gautam@example.com"

What Does --global Mean?

The --global flag means this setting applies to all your Git projects on this computer. You only need to do this once.

Without --global, the setting would only apply to one specific project.

Checking Your Settings

To confirm everything was saved correctly, type:

git config --list

You will see a list of settings. Look for these two lines:

user.name=Gian
user.email=gautam@example.com

If you see your name and email, you are all set!

Example: Our Team Sets Up Git

Gian's computer:

git config --global user.name "Gian"
git config --global user.email "gian@doraemon.com"

Nobita's computer:

git config --global user.name "Nobita"
git config --global user.email "nobita@doraemon.com"

Suneo's computer:

git config --global user.name "Suneo"
git config --global user.email "suneo@doraemon.com"

Now when each person makes changes, Git will record who made them.

Key Concepts

Before learning Git commands, you need to understand some important terms. These words will appear constantly when working with Git.

Repository

A repository (also called a repo) is a project folder that Git is tracking. It contains all your files plus a hidden folder called .git where Git stores the history.

Example

Gian creates a folder called school-project for the project. When he tells Git to track this folder, it becomes a repository.

Commit

A commit is a saved snapshot of your project at one specific point in time. When you make a commit, you are telling Git: "Save this exact version forever."

Every commit has:

  • A unique ID - A long code like a1b2c3d4 that identifies this snapshot
  • A message - A short description you write explaining what changed
  • Author - Who made the commit
  • Date - When it was made

Good commit messages look like:

  • "Add login page"
  • "Fix calculation error in score"
  • "Update homepage design"

Branch

A branch is like a parallel copy of your project where you can experiment without affecting the main code.

Your main project lives on the main branch (sometimes called master). If you want to try a new feature, you create a new branch, work there, and if it works, you bring it back to main. If it fails, you just delete the branch.

main branch:    A --- B --- C --- D
                         \
feature branch:           E --- F

Merge

When you finish working on a branch and want to add your work back into the main project, you merge the branches together. Git combines the changes automatically.

Remote

A remote is a version of your repository stored somewhere else - usually on GitHub. The remote acts as your online backup and allows you to share your code with others.

By convention, the main remote is named origin.

Clone

Cloning means downloading a complete copy of a repository from GitHub to your computer, including all files and the entire history of changes.

Example: Our Team's Repository

Gian creates a repository called school-project. This repository has:

  • A main branch where the working code lives
  • Multiple commits showing the history of changes
  • A remote on GitHub where the code is backed up

Nobita creates a feature-login branch to work on the login page without breaking the main code.

Suneo clones the repository from GitHub to his computer so he can work on it too.

The Three Stages of Git

This is the MOST IMPORTANT concept in Git. Understanding these three stages will make everything else make sense.

Overview

Every file in your Git project exists in one of three states at any given time:

1. Working Directory  →  2. Staging Area  →  3. Repository
   (where you edit)      (files ready to save)  (permanently saved)

Stage 1: Working Directory

This is simply the folder on your computer where your project files live. When you create a new file, edit an existing one, or delete something, those changes happen in the Working Directory.

Git can see that changes happened, but it has not saved anything yet.

Think of it like this:

Writing on a whiteboard - the words are there for now, but they are not permanent. Anyone could erase them.

Example: Gian creates a new file called notes.txt and writes some text in it. This file is in the Working Directory.

# Check what changed
git status

Untracked files:
  notes.txt

Stage 2: Staging Area

The Staging Area (also called the Index) is a preparation zone. Before saving a snapshot, you pick and choose which changed files to include.

This gives you fine control - maybe you changed 5 files but only want to save 2 of them in this snapshot.

Think of it like this:

Preparing an envelope to mail - you have decided what goes inside, but you have not sealed and sent it yet.

Example: Gian decides to stage notes.txt:

# Add file to staging area
git add notes.txt

git status

Changes to be committed:
  new file: notes.txt

Stage 3: Repository

Once you commit, Git permanently records a snapshot of all staged files into the repository. This is stored inside a hidden folder called .git inside your project.

From this point on, this version exists forever in Git's memory - you can always come back to it.

Think of it like this:

Sealing the envelope and locking it in a safe. It is permanent and cannot be changed.

Example: Gian commits the staged file:

# Commit with a message
git commit -m "Add notes file"

[main a1b2c3d] Add notes file
 1 file changed, 1 insertion(+)

Complete Example

Let's see all three stages in action with our cricket team:

# Gian creates a new file (Working Directory)
# He opens a text editor and creates: score.txt
# Content: "Team A: 250 runs"

# Check status
git status
Untracked files:
  score.txt

# Move to Staging Area
git add score.txt

# Check status again
git status
Changes to be committed:
  new file: score.txt

# Commit to Repository
git commit -m "Add initial score"
[main b2c3d4e] Add initial score
 1 file changed, 1 insertion(+)

# Check status one more time
git status
nothing to commit, working tree clean

Working with Multiple Files

You can stage multiple files at once:

# Stage specific files
git add file1.txt file2.txt file3.txt

# Stage ALL changed files
git add .

# Then commit them all together
git commit -m "Add multiple files"

Important Rule

You must always do git add before git commit. If you skip the add step, nothing gets committed because the Staging Area is empty.

Basic Git Commands

These are the essential commands you will use every day when working with Git.

git init - Start Tracking a Folder

This command turns a normal folder into a Git repository. Run it once inside the folder you want to track.

# Navigate to your project folder
cd school-project

# Initialize Git
git init

Initialized empty Git repository in /school-project/.git/

Git creates a hidden .git folder inside. This is Git's database - do not delete or edit it manually.

git status - Check What Changed

This is the command you will use most often. It shows you which files have been changed, which are staged, and which are not being tracked yet.

git status

git add - Stage Your Changes

Move files from Working Directory to Staging Area.

# Stage one specific file
git add filename.txt

# Stage multiple files
git add file1.txt file2.txt

# Stage all changed files
git add .

git commit - Save a Snapshot

Save all staged files permanently with a message describing what changed.

git commit -m "Your message here"

Write your commit messages in present tense. Good examples:

  • "Add login page"
  • "Fix score calculation bug"
  • "Update homepage design"

git log - View History

See a list of all commits (snapshots) you have made.

# Full detailed history
git log

# Compact one-line version (easier to read)
git log --oneline

a1b2c3d Add login page
9e8f7g6 Fix score bug
5h4i3j2 Initial commit

Press Q to quit the log view.

git diff - See What Changed

Shows you the actual lines that were added or removed in your files.

git diff

Complete Example: Gian's First Day

# Day 1: Create project folder
mkdir school-project
cd school-project

# Initialize Git
git init

# Create a file (using text editor, save as readme.txt)
# Content: "School Project Tracker Project"

# Check status
git status
Untracked files:
  readme.txt

# Stage the file
git add readme.txt

# Commit
git commit -m "Add readme file"

# View history
git log --oneline
a1b2c3d Add readme file

Quick Reference Table

Command What It Does
git init Start Git tracking in a folder
git status Check which files changed
git add filename Stage one specific file
git add . Stage all changed files
git commit -m "message" Save a permanent snapshot
git log View commit history
git log --oneline View compact history
git diff See line-by-line changes

gitignore and gitkeep

What is .gitignore?

Sometimes you have files you do NOT want Git to track. For example:

  • Password files
  • Temporary files
  • Large data files
  • Files that can be regenerated

The .gitignore file tells Git which files to ignore.

Why Ignore Files?

Example from Our Team

Gian has a file called passwords.txt with database passwords. He does NOT want this file on GitHub where everyone can see it. He also has a temp folder with temporary files that are not important.

Creating .gitignore

Create a file named .gitignore in your project root folder (note the dot at the beginning).

Inside this file, list the files and folders you want Git to ignore:

# Ignore password file
passwords.txt

# Ignore entire temp folder
temp/

# Ignore all log files
*.log

# Ignore config folder
config/

Complete Example

# Gian creates .gitignore file
# (Use text editor to create file named: .gitignore)
# Add these lines:

passwords.txt
temp/
*.log
node_modules/

# Now stage and commit the .gitignore file itself
git add .gitignore
git commit -m "Add gitignore file"

# Git will now ignore those files
git status
nothing to commit, working tree clean
# (passwords.txt is not shown even though it exists)

Common Patterns

Pattern What It Ignores
filename.txt Specific file
folder/ Entire folder
*.log All files ending with .log
temp* All files starting with temp
!important.log Exception - do NOT ignore this file

What is .gitkeep?

Git does not track empty folders. But sometimes you need to keep an empty folder in your repository.

Problem: Suneo creates an uploads folder where users will upload files later. The folder is empty now, but he wants it in the repository.

Solution: Create an empty file called .gitkeep inside the folder.

Example

# Suneo creates uploads folder
mkdir uploads

# Create .gitkeep file inside it
# (Use text editor to create empty file: uploads/.gitkeep)

# Now Git can track the folder
git add uploads/.gitkeep
git commit -m "Add uploads directory"

Note

.gitkeep is not a Git feature - it is just a convention. You could name the file anything, but .gitkeep is the standard that everyone uses.

Undoing Changes

Everyone makes mistakes. Git provides several ways to undo changes depending on where the changes are (Working Directory, Staging Area, or Repository).

Unstaging Files

You staged a file by mistake and want to unstage it (move it back to Working Directory).

# Nobita accidentally staged wrong file
git add wrong-file.txt

# Unstage it
git restore --staged wrong-file.txt

# Or unstage ALL files
git restore --staged .

Discarding Changes in Working Directory

You made changes to a file but want to throw them away and go back to the last committed version.

# Discard changes in one file
git restore filename.txt

# Discard ALL changes
git restore .

Warning

git restore permanently deletes your uncommitted changes. They cannot be recovered! Use this command carefully.

Undoing the Last Commit

You made a commit but realized it was too early or had a mistake.

Option 1: Keep Changes (Soft Reset)

# Undo last commit, keep changes in staging area
git reset --soft HEAD~1

Option 2: Unstage Changes (Mixed Reset)

# Undo last commit, keep changes but unstage them
git reset HEAD~1

Option 3: Delete Everything (Hard Reset)

# Undo last commit and DELETE all changes
git reset --hard HEAD~1

Danger

git reset --hard permanently deletes your changes. Only use this if you are absolutely sure!

Going Back to a Specific Commit

# View commit history
git log --oneline
a1b2c3d Latest commit
9e8f7g6 Previous commit
5h4i3j2 Old commit
1k2l3m4 Very old commit

# Go back to specific commit
git reset 5h4i3j2

Reverting a Commit (Safe Way)

Instead of deleting history, create a NEW commit that undoes a previous commit. This is safer for code already pushed to GitHub.

# Safely undo a specific commit
git revert 9e8f7g6

# Git opens editor for commit message
# Save and close - a new "revert" commit is created

Example: Nobita's Mistake

# Nobita edits score.txt
# He adds wrong data

# He stages it
git add score.txt

# Wait! He realizes the mistake
# Unstage the file
git restore --staged score.txt

# Discard the wrong changes
git restore score.txt

# File is back to last committed version

Best Practice

  • Use git revert for commits already pushed to GitHub
  • Use git reset only for local commits not yet pushed
  • Never use --hard unless you are absolutely sure

Understanding Branches

Branches are one of Git's most powerful features. They let you work on new features without breaking the main code.

Why Use Branches?

Imagine this scenario:

  • Gian's main code is working perfectly
  • Nobita wants to add a new login feature
  • While Nobita is working, the login feature is half-done and broken
  • If Nobita works directly on the main code, he will break it for everyone

Solution: Nobita creates a separate branch to work on the login feature. The main code stays working while he experiments.

Visual Representation

main branch (working code):
  A --- B --- C --- D

Nobita creates feature-login branch:
  A --- B --- C --- D
               \
                E --- F (Nobita's work)

When done, merge back to main:
  A --- B --- C --- D --- G (merged)
               \         /
                E ----- F

Creating a Branch

# Create a new branch
git branch feature-login

# Switch to that branch
git checkout feature-login

# Or do both in one command
git checkout -b feature-login

Switched to a new branch 'feature-login'

Listing Branches

git branch

  main
* feature-login

# The * shows which branch you are currently on

Switching Branches

# Switch to main branch
git checkout main

# Switch to feature branch
git checkout feature-login

Complete Example: Nobita's Login Feature

# Nobita is on main branch
git checkout main

# Create and switch to new branch
git checkout -b feature-login

# Nobita creates login.txt file
# Content: "Login Page Code"

# Stage and commit on feature branch
git add login.txt
git commit -m "Add login page"

# Make more changes
# Edit login.txt, add more code

git add login.txt
git commit -m "Complete login functionality"

# View commits on this branch
git log --oneline
f3e2d1c Complete login functionality
a9b8c7d Add login page
5h4i3j2 (main) Previous main commit

# Switch back to main - login.txt disappears!
git checkout main

# Switch back to feature-login - login.txt reappears!
git checkout feature-login

Deleting a Branch

# Delete a branch (after merging)
git branch -d feature-login

# Force delete (even if not merged)
git branch -D feature-login

Our Team's Branching Strategy

main branch (Gian manages):
  A --- B --- C

Nobita's branches:
  - feature-login
  - feature-ui-design

Suneo's branches:
  - feature-database
  - feature-api

Everyone works independently without breaking main code!

Merging Branches

When you finish working on a feature branch, you need to bring those changes back into the main branch. This is called merging.

How Merging Works

Merging combines the changes from one branch into another branch.

Before merge:
main:    A --- B --- C
              \
feature:       D --- E

After merge:
main:    A --- B --- C --- F (merge commit)
              \           /
feature:       D ------- E

Steps to Merge

  1. Switch to the branch you want to merge INTO (usually main)
  2. Run the merge command
  3. Git combines the changes automatically

Example: Merging Nobita's Login Feature

# Nobita finished working on feature-login branch
# Now Gian will merge it into main

# Step 1: Switch to main branch
git checkout main

# Step 2: Merge the feature branch
git merge feature-login

Updating a1b2c3d..f3e2d1c
Fast-forward
 login.txt | 10 ++++++++++
 1 file changed, 10 insertions(+)

# Step 3: Delete the feature branch (optional)
git branch -d feature-login

Types of Merges

Fast-Forward Merge

This happens when the main branch has not changed since you created the feature branch. Git simply moves the main branch pointer forward.

Before:
main:    A --- B
              \
feature:       C --- D

After (fast-forward):
main:    A --- B --- C --- D

Three-Way Merge

This happens when both branches have new commits. Git creates a new "merge commit" that combines both.

Before:
main:    A --- B --- C
              \
feature:       D --- E

After (three-way):
main:    A --- B --- C --- F (merge commit)
              \           /
feature:       D ------- E

Complete Team Example

# Nobita creates feature branch
git checkout -b feature-login

# Nobita works and commits
# Creates login.txt
git add login.txt
git commit -m "Add login page"

# Meanwhile, Suneo also creates a branch
git checkout main
git checkout -b feature-database

# Suneo works and commits
# Creates database.txt
git add database.txt
git commit -m "Add database connection"

# Gian merges Nobita's work first
git checkout main
git merge feature-login
Fast-forward merge successful

# Then merges Suneo's work
git merge feature-database
Merge made by the 'recursive' strategy

# Now main has both features!

Merge Conflicts

A merge conflict happens when two people edit the SAME lines in the SAME file. Git does not know which version to keep, so it asks you to decide.

When Do Conflicts Happen?

Conflict Scenario

Gian edits score.txt line 1:

Total Score: 100

Nobita also edits score.txt line 1:

Total Score: 200

When they try to merge, Git says: "I don't know which one to keep!"

Creating a Conflict (Example)

# Gian on main branch
git checkout main
# Edit score.txt, line 1: "Total Score: 100"
git add score.txt
git commit -m "Set score to 100"

# Nobita on feature branch
git checkout -b feature-scoring
# Edit score.txt, line 1: "Total Score: 200"
git add score.txt
git commit -m "Set score to 200"

# Try to merge
git checkout main
git merge feature-scoring

Auto-merging score.txt
CONFLICT (content): Merge conflict in score.txt
Automatic merge failed; fix conflicts and then commit the result.

What a Conflict Looks Like

Git modifies the conflicted file to show both versions:

<<<<<<< HEAD
Total Score: 100
=======
Total Score: 200
>>>>>>> feature-scoring

Understanding Conflict Markers

  • <<<<<<< HEAD - Your current branch (main)
  • ======= - Separator between the two versions
  • >>>>>>> feature-scoring - The branch you are merging

Resolving the Conflict

Step 1: Open the File

Open score.txt in your text editor. You will see the conflict markers.

Step 2: Decide What to Keep

You have three options:

  • Keep your version (100)
  • Keep their version (200)
  • Keep both or write something new

Let's say Gian decides to keep Nobita's version (200). Edit the file to:

Total Score: 200

Important: Remove ALL conflict markers (<<<<<<<, =======, >>>>>>>)

Step 3: Stage the Resolved File

git add score.txt

Step 4: Complete the Merge

git commit -m "Resolve merge conflict - use score 200"

[main a1b2c3d] Resolve merge conflict - use score 200

Checking for Conflicts

# See which files have conflicts
git status

Unmerged paths:
  both modified:   score.txt

Aborting a Merge

If you want to cancel the merge and start over:

git merge --abort

Real Team Conflict Example

# Nobita and Suneo both edited homepage.txt

# Nobita's version (line 1):
School Project Tracker - Login System

# Suneo's version (line 1):
School Project Tracker

# After merge attempt, file shows:
<<<<<<< HEAD
School Project Tracker - Login System
=======
School Project Tracker
>>>>>>> feature-backend

# Gian decides to combine both ideas:
School Project Tracker - Login System

# Resolve:
git add homepage.txt
git commit -m "Merge titles from both features"

Conflict Resolution Steps Summary

  1. Run git status to see conflicted files
  2. Open each file and look for <<<<<<< markers
  3. Decide which code to keep
  4. Remove all conflict markers
  5. Save the file
  6. git add filename
  7. git commit

What is GitHub?

Now that you understand Git, let's learn about GitHub - the place where developers store and share their code online.

Git vs GitHub

Git GitHub
Software on your computer Website (github.com)
Tracks changes locally Stores code online
Works offline Requires internet
Free and open source Free for public projects

Why Use GitHub?

  • Backup: Your code is safely stored in the cloud
  • Collaboration: Multiple people can work on the same project
  • Sharing: Share your code with the world
  • Portfolio: Show your work to employers and colleges
  • Open Source: Contribute to projects used by millions

GitHub acts as the central hub where everyone syncs their work.

Key GitHub Features

  • Repositories: Store your projects
  • Issues: Track bugs and feature requests
  • Pull Requests: Propose changes to code
  • Actions: Automate tasks
  • Pages: Host websites for free

Creating a GitHub Account

Let's create your GitHub account. It is free and takes only a few minutes.

Sign Up Steps

  1. Open your web browser and go to https://github.com
  2. Click the Sign up button in the top right corner
  3. Enter your email address and click Continue
  4. Create a strong password (at least 8 characters) and click Continue

Push and Pull

These are the two most important commands for working with GitHub.

git push - Upload Your Changes

After you make commits on your computer, use git push to upload them to GitHub.

# First time pushing (sets upstream)
git push -u origin main

# Every time after that
git push

What does -u mean?

The -u flag (short for --set-upstream) tells Git that from now on, when you type git push, it should automatically push to origin main. You only need -u the first time.

git pull - Download Latest Changes

If the code on GitHub has been updated (by you on another computer or by a teammate), download those changes with git pull.

git pull

remote: Enumerating objects: 5, done.
Updating a1b2c3d..f3e2d1c
Fast-forward
 newfile.txt | 1 +
 1 file changed, 1 insertion(+)

Daily Workflow

This is the sequence every professional developer follows:

# 1. Always start by downloading latest code
git pull

# 2. Make your changes in text editor...

# 3. Check what changed
git status

# 4. Stage all changes
git add .

# 5. Commit with a message
git commit -m "Describe what you changed"

# 6. Upload to GitHub
git push

Golden Rule

Always run git pull before you start working. This keeps your local copy in sync with GitHub and prevents conflicts.

git fetch vs git pull

git fetch: Download changes but don't merge them yet

git fetch origin
# View what changed
git log origin/main
# Merge when ready
git merge origin/main

git pull: Fetch + Merge in one command

git pull
# Same as: git fetch + git merge

Git Stash

Stash temporarily saves your uncommitted changes so you can work on something else.

When to Use Stash?

Scenario

Nobita is working on a feature. Suddenly, Gian asks him to fix an urgent bug. Nobita's current work is not ready to commit. What should he do?

Solution: Stash the current work, fix the bug, then bring back the stashed work.

Stashing Changes

# Save current changes
git stash

Saved working directory and index state

# Your working directory is now clean
git status
nothing to commit, working tree clean

Viewing Stashes

git stash list

stash@{0}: WIP on feature-login: a1b2c3d Add login
stash@{1}: WIP on main: 9e8f7g6 Fix bug

Applying Stashed Changes

# Apply most recent stash
git stash apply

# Apply specific stash
git stash apply stash@{1}

# Apply and remove from stash list
git stash pop

Complete Example

# Nobita is working on feature
# Edits login.txt but not ready to commit

# Urgent bug needs fixing!
# Stash current work
git stash

# Switch to main branch
git checkout main

# Fix the bug
git add bugfix.txt
git commit -m "Fix urgent bug"
git push

# Go back to feature branch
git checkout feature-login

# Bring back stashed work
git stash pop

# Continue working on feature

Deleting Stashes

# Delete specific stash
git stash drop stash@{0}

# Delete all stashes
git stash clear

Git Rebase

Rebase is an alternative to merge. It creates a cleaner, linear history.

Rebase vs Merge

MERGE:
main:    A --- B --- C --- M (merge commit)
              \           /
feature:       D ------- E

REBASE:
main:    A --- B --- C
                         \
feature:                  D' --- E'
(commits moved to end of main)

How to Rebase

# On feature branch
git checkout feature-login

# Rebase onto main
git rebase main

Successfully rebased and updated refs/heads/feature-login.

When to Use Rebase?

  • To keep a clean, linear history
  • Before merging a feature branch
  • To update your branch with latest main

Warning

Never rebase commits that have been pushed to GitHub and shared with others. Only rebase local commits.

Example

# Nobita's feature branch is behind main
git checkout feature-login

# Update with latest main
git rebase main

# Now feature branch has latest main changes
# And commits are in a clean line
